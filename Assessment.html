Executive Summary

XDBC is a well-architected Design by Contract framework using TypeScript decorators. The project demonstrates strong fundamental design — clean class hierarchy, consistent decorator patterns, and a rich contract library of 16 validators. However, the assessment reveals several critical correctness bugs (parameter ordering, null-check logic), insufficient test coverage (decorators are entirely untested), and security gaps that should be addressed before production use.

1. Architecture & Design

Strengths

  • Consistent decorator pattern: all contracts expose PRE, POST, and INVARIANT static factories
  • Clear class hierarchy with DBC as base and specialized contracts extending it
  • Separation of concerns between static factories (contract definition) and instance methods (validation logic)
  • Plugin architecture supporting multiple DBC instances via namespace resolution
  • Extensible contract objects with composable check() method signature

Issues

  • Global state coupling — WaXCode.DBC namespace on globalThis makes isolated testing difficult
  • Inconsistent abstraction — some contracts have check() instance methods, others only static checkAlgorithm()
  • No base factory — adding new contracts requires copy-pasting PRE/POST/INVARIANT boilerplate

2. Code Quality

Type Safety

  • 40+ any annotations suppressed with biome-ignore comments
  • COMPARISON.checkAlgorithm() has untyped parameters
  • Unsafe casts: (this as any).constructor in DBC.ts

Error Messages

  • Typo: "has to to be" in COMPARISON.ts, TYPE.ts, EQ.ts
  • Object interpolation produces [object Object] in IF.ts error messages
  • User values interpolated directly without escaping

Duplication

  • GREATER, LESS, GREATER_OR_EQUAL, LESS_OR_EQUAL are 95% identical code
  • PRE/POST/INVARIANT factory code repeated across all 16 contract classes
  • Error formatting duplicated 3 times in DBC.ts

3. Critical Bugs

CRITICAL Bug 1 — COMPARISON.POST() parameter swap

src/DBC/COMPARISON.ts — POST decorator factory

Arguments to checkAlgorithm are passed in the wrong order — equivalent and equalityPermitted are swapped. Postcondition checks compare against a boolean value and use a number as the equality flag.

// Current (broken):
COMPARISON.checkAlgorithm(value, equalityPermitted, equivalent, invert);

// Should be:
COMPARISON.checkAlgorithm(value, equivalent, equalityPermitted, invert);

CRITICAL Bug 2 — JSON_OP null check logic error

src/DBC/JSON.OP.ts

Incorrect operator precedence causes the null branch to never trigger for null inputs.

// Current (broken):
if (toCheck === undefined || null)

// Should be:
if (toCheck === undefined || toCheck === null)

CRITICAL Bug 3 — DIFFERENT.PRE() hint/dbc swapped

src/DBC/EQ/DIFFERENT.ts

Calls EQ.PRE(equivalent, true, path, dbc, hint) but EQ.PRE expects (equivalent, invert, path, hint, dbc). The hint and dbc parameters are transposed.

CRITICAL Bug 4 — GREATER_OR_EQUAL, LESS, LESS_OR_EQUAL parameter swap

src/DBC/COMPARISON/GREATER_OR_EQUAL.ts, LESS.ts, LESS_OR_EQUAL.ts

All three pass path, dbc, hint to COMPARISON.PRE() which expects path, hint, dbc.

HIGH Bug 5 — getDBC() caches undefined

src/DBC.ts

If resolveDBCPath() returns undefined, the result is cached. Calls to getDBC().executionSettings will throw a null reference error.

9. Strengths

  • Rich contract library: 16 contracts covering types, equality, regex, JSON, arrays, instances, conditionals, and Zod schemas
  • Decorator ergonomics: Non-invasive validation preserving clean method signatures
  • Detailed error context: Infringement messages include class name, method name, parameter index, and violation details
  • Configuration flexibility: Enable/disable preconditions, postconditions, and invariants independently
  • Path resolution: Dot notation, array indices, method calls, and HTML attributes supported
  • Type-safe static checks: tsCheck methods for imperative use outside decorators
  • Standard regex library: Ready-to-use patterns for email, URL, BCP47, date, CSS selectors
  • Modern tooling: Current versions of TypeScript, Webpack, Biome, Jest

10. Recommendations

PRIORITY 1 — Critical   Estimated: 3 hours

  1. Fix parameter ordering bugs in COMPARISON.POST, DIFFERENT.PRE, GREATER_OR_EQUAL.PRE, LESS.PRE, LESS_OR_EQUAL.PRE
  2. Fix null check in JSON_OP.checkAlgorithm
  3. Add zod to package.json dependencies
  4. Add null guard after getDBC() calls
  5. Remove console.log from ZOD.ts

PRIORITY 2 — High   Estimated: 6 hours

  1. Write decorator integration tests for PRE, POST, INVARIANT, and ParamvalueProvider
  2. Fix test describe names in OR, REGEX, TYPE, JSON.Parse test files
  3. Fix GREATER.test.ts import path
  4. Standardize null/undefined handling across all contracts
  5. Fix "has to to be" typo in error messages

PRIORITY 3 — Medium   Estimated: 12 hours

  1. Extract base factory for PRE/POST/INVARIANT to eliminate duplication
  2. Add path validation in DBC.resolve() (block __proto__, validate syntax)
  3. Implement cache eviction (LRU or bounded size) for static caches
  4. Sanitize error message interpolation to prevent XSS in web contexts
  5. Add check() instance methods to TYPE, DEFINED, UNDEFINED for API consistency
  6. Reduce any usage — create a strict Contract interface
  7. Add CI/CD workflow (GitHub Actions) with lint + test + build
  8. Remove @types/express from devDependencies
